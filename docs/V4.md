# MTProto Proxy со своим доменом, FakeTLS и спонсорским каналом (2026)

Финальная рабочая инструкция. Прокси маскируется под твой собственный домен (`your.domain`) с **настоящим Let's Encrypt сертификатом**, поддерживает спонсорский канал через @MTProxybot, и для DPI выглядит как обычный сайт на nginx.

## Архитектура

```
[ИНТЕРНЕТ] ──→ your.domain
              │
              ├── :80   → Caddy (renew LE certs, HTTP→HTTPS redirect)
              ├── :443  → Caddy (LE cert для your.domain, decoy "OK"-страница)
              └── :853  → alexbers (FakeTLS, забирает LE cert с Caddy:443
                                    при старте, обрабатывает MTProxy-трафик)
```

Зачем такая схема. Сам alexbers при старте подключается к `MASK_HOST:443`, скачивает оттуда сертификат и использует его для FakeTLS-маскировки. Если запустить alexbers «голым» под собственным доменом — он пытается стянуть cert у себя же → получает пустой ответ или left-over cert от соседнего сервиса. **Caddy решает это**: на 443 поднимается настоящий TLS-сервер с LE-сертификатом для домена, alexbers корректно его получает.

## Шаг 0. Что нужно

| Требование | Зачем |
|---|---|
| VPS Ubuntu 22.04+ / Debian 12 с root | Хост |
| Зарубежный хостер с чистой маршрутизацией к Telegram | Без этого медиа не грузится — см. Шаг 1 |
| Свой домен (или сабдомен), к нему есть DNS-доступ | Для A-записи и LE-валидации |
| Открытые **80, 443, 853** TCP в фаерволе хостера | 80/443 — Caddy, 853 — alexbers |

Проверенные хостеры с чистым роутингом: **Hostkey**, Hetzner, UpCloud, TransIP, CDN77 — все европейские.

## Шаг 1. Pre-flight: проверка маршрута к Telegram DC

**До установки чего-либо**. Симптом плохой маршрутизации — текст ходит, медиа висит. Хостеры режут точечно (одни IP в подсети живые, другие нет):

```bash
for ip in 149.154.167.50 149.154.175.50 149.154.175.100 \
          149.154.165.109 91.108.4.136 91.108.56.183 \
          91.108.56.100 91.108.56.130; do
  timeout 3 nc -zv $ip 443 2>&1 | tail -1
  timeout 3 nc -zv $ip 8888 2>&1 | tail -1
done
```

**ВСЕ** должны быть `succeeded`. Если хоть один в timeout — этот хостер не годится. Особенно критичны IP с портом **8888** — это middle-proxies, через которые регистрируется ad-tag (без них спонсорский канал работать не будет).

## Шаг 2. Подготовка сервера

```bash
sudo apt update && sudo apt install -y docker.io docker-compose-v2 git curl
sudo systemctl enable --now docker
```

Если `docker compose version` (v2 с пробелом) выдаёт ошибку — у тебя только v1, везде ниже подставляй `docker-compose` через дефис.

Открой порт в ufw (если активен):
```bash
sudo ufw allow 80/tcp 443/tcp 853/tcp
```

В панели хостера (Security Groups / Firewall) разреши **входящий TCP 80, 443, 853**.

## Шаг 3. DNS-запись

В админке твоего регистратора домена создай **A-запись**:
```
your.domain.    A    IP_ТВОЕГО_VPS
```

Подожди 1–5 минут, проверь:
```bash
dig +short your.domain
# должен вернуть IP твоего VPS
```

Если возвращает не тот IP или ничего — TTL ещё не истёк, либо запись настроена неверно. Не идём дальше пока не работает.

## Шаг 4. Структура проекта

```bash
mkdir -p /root/mtproxy && cd /root/mtproxy
git clone -b stable https://github.com/alexbers/mtprotoproxy.git src
```

В каталоге `/root/mtproxy/` будут лежать 4 файла:
- `src/` — свежий исходник alexbers (для сборки образа Docker)
- `Caddyfile` — конфиг Caddy
- `docker-compose.yml` — оба сервиса
- `config.py` — конфиг alexbers

---

## Шаг 5. Caddyfile

`/root/mtproxy/Caddyfile`:
```caddyfile
your.domain {
	respond "OK" 200
	header Server "nginx"
}
```

Что делает: Caddy при старте автоматически получает LE-сертификат через ACME (TLS-ALPN-01 challenge), серверит «200 OK» с заголовком `Server: nginx` для любого HTTPS-запроса. Для DPI выглядит как обычный nginx, отдающий легковесную страничку. Именно этот сертификат потом утянет alexbers.

---

## Шаг 6. Базовый секрет

Базовый секрет — 32 hex-символа (`0-9`, `a-f`). Это **«ключ пользователя»** для прокси.

```bash
head -c 16 /dev/urandom | xxd -ps
# например: 89bea1ac525b5c7e58939d18a26e2ae5
```

**Сохрани**. Этот секрет:
- Используется в config.py как идентификатор пользователя (USERS dict)
- Отдаётся боту @MTProxybot для регистрации (Шаг 11)
- Не раздаётся в чистом виде пользователям — они получают его в составе FakeTLS-секрета (см. Шаг 14)

---

## Шаг 7. config.py для alexbers

`/root/mtproxy/config.py`:
```python
PORT = 853

USERS = {
    "user1": "ВСТАВЬ_ТВОЙ_БАЗОВЫЙ_СЕКРЕТ_ИЗ_ШАГА_6",
}

TLS_DOMAIN = "your.domain"

# AD_TAG раскомментируй после Шага 11:
# AD_TAG = "..."
```

Параметры:
- `PORT = 853` — порт alexbers (DoT, выглядит легитимно). Можно использовать другой высокий порт.
- `USERS` — словарь {имя: 32-hex секрет}. Можно несколько для статистики по людям.
- `TLS_DOMAIN` — твой домен для FakeTLS-маскировки. **Должен совпадать** с тем что в Caddyfile.

## Шаг 8. docker-compose.yml

`/root/mtproxy/docker-compose.yml`:
```yaml
services:
  caddy:
    image: caddy:2
    container_name: mtproxy-caddy
    restart: unless-stopped
    network_mode: host
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
      - caddy_config:/config

  alexbers:
    build: ./src
    container_name: mtproto-final
    restart: unless-stopped
    network_mode: host
    volumes:
      - ./config.py:/home/tgproxy/config.py:ro
    depends_on:
      - caddy

volumes:
  caddy_data:
  caddy_config:
```

Ключевые моменты:
- `network_mode: host` — оба контейнера биндятся напрямую на хост, без посредника `docker-proxy` (быстрее, проще диагностика).
- `caddy_data` — Docker volume где хранится **приватный ключ LE-сертификата**. Не теряй его и не публикуй наружу.
- `volumes: - ./config.py:/home/tgproxy/config.py:ro` — путь монтирования внутри контейнера обязательно `/home/tgproxy/`, не `/home/tmuser/` (старые гайды!).
- `depends_on: - caddy` — alexbers стартует после Caddy, чтобы успеть забрать сертификат.

## Шаг 9. Запуск Caddy и получение LE-сертификата

Сначала только Caddy, чтобы убедиться что cert получен корректно:

```bash
cd /root/mtproxy
docker-compose up -d caddy
sleep 15
docker-compose logs --tail 30 caddy | grep -E "obtain|certificate"
```

Должно быть:
```
"msg":"certificate obtained successfully","identifier":"your.domain","issuer":"acme-v02.api.letsencrypt.org-directory"
```

Проверь что cert настоящий:
```bash
echo | openssl s_client -connect 127.0.0.1:443 -servername your.domain 2>&1 | \
  openssl x509 -noout -subject -issuer -dates
```

Должно отдать:
```
subject=CN = your.domain
issuer=C = US, O = Let's Encrypt, CN = E7
notBefore=...
notAfter=...
```

Если LE не выдаёт cert — проверь:
- DNS правильно резолвит (`dig +short your.domain`)
- Порт 80 открыт снаружи (LE использует HTTP-01 или TLS-ALPN-01 challenge)
- Нет других сервисов на 80/443

## Шаг 10. Запуск alexbers

```bash
docker-compose up -d --build alexbers
sleep 5
docker-compose logs --tail 15 alexbers
```

В логах должно быть:
```
user1: tg://proxy?server=...&port=853&secret=...
user1: tg://proxy?server=...&port=853&secret=dd...
user1: tg://proxy?server=...&port=853&secret=ee...74672e6c696d626f76706e2e7275
Found uvloop, using it for optimal performance
Got cert from the MASK_HOST your.domain, its length is 2060
```

**Главный признак успеха** — `Got cert from the MASK_HOST your.domain` (а не google.com, www.google.com или что-то другое).

Если вместо этого видишь `default secret`/`default TLS_DOMAIN` — конфиг не примонтировался: `docker exec mtproto-final cat /home/tgproxy/config.py`.

Если предупреждение `sent some TLS record before certificate record` — Caddy формирует TLS-ответ чуть нестандартно. Не блокер, проксь работает; в случае проблем с DPI можно заменить Caddy на nginx (TLS-фрейминг у nginx ближе к «классическому»).

## Шаг 11. Регистрация прокси в @MTProxybot (для спонсорского канала)

Если канал не нужен — пропусти этот и следующие два шага.

В Telegram напиши [@MTProxybot](https://t.me/MTProxybot):

```
/newproxy
```

Бот спросит `host:port` — отправь твой домен и порт alexbers:
```
your.domain:853
```

Бот спросит секрет — отправь **базовый 32-hex** (тот, что из Шага 6, **БЕЗ префикса `ee`** и **БЕЗ домена**):
```
89bea1ac525b5c7e58939d18a26e2ae5
```

Бот ответит:
```
Success! Your proxy has been successfully registered. You can now pass this proxy tag to the software you are using: 1bf1af94d53231026da2b3f2dc731b20.
```

**`1bf1af94d53231026da2b3f2dc731b20`** — это **AD_TAG**. Сохрани.

## Шаг 12. КРИТИЧЕСКИ ВАЖНО: привязка канала

Регистрация в `/newproxy` создаёт запись прокси, но **не привязывает к нему канал**. Канал нужно привязать **отдельным шагом**:

```
/myproxies
```

Бот покажет список твоих прокси. Найди запись `your.domain:853` (новая), тапни на неё. Появятся кнопки:
- **`Set promoted channel`** ← вот эту
- `Suspend Promotion`
- `Delete`

Жми `Set promoted channel`, отправь:
```
@your_channel
```
или
```
t.me/your_channel
```

Бот подтвердит:
```
Promoted channel: Your Channel (https://t.me/your_channel).
```

**Без этого шага канал НЕ ПОЯВИТСЯ у пользователей**, даже если AD_TAG корректно настроен в alexbers. Проверено эмпирически.

Если у тебя есть **старые** записи в `/myproxies` от прошлых проксей — удалить их через `Delete`, чтобы счётчики не путались.

## Шаг 13. Добавление AD_TAG в config.py и перезапуск

Открой `config.py`:
```bash
nano /root/mtproxy/config.py
```

Раскомментируй `AD_TAG` и подставь полученный из бота тег:
```python
AD_TAG = "1bf1af94d53231026da2b3f2dc731b20"
```

Перезапусти **только alexbers** (Caddy трогать не нужно):
```bash
cd /root/mtproxy
docker-compose restart alexbers
```

В логах после рестарта alexbers подключится к Telegram middle-proxies (порт 8888) и начнёт регистрировать ad-tag. Иногда видно:
```
Unable to connect to the Telegram server X (...)
```
Это нормально, если потом alexbers переключается на другой middle-proxy. Проверить что соединения есть:
```bash
docker exec mtproto-final cat /proc/net/tcp | awk '$4=="01"' | grep ":22B8"
# 22B8 = 8888 hex; должно быть несколько ESTABLISHED-строк
```

## Шаг 14. Раздаваемая ссылка

Финальная ссылка для пользователей собирается из трёх частей:

```
https://t.me/proxy?server=your.domain&port=853&secret=ee[BASE_SECRET][HEX_DOMAIN]
```

Где:
- `[BASE_SECRET]` — твой 32-hex секрет из Шага 6
- `[HEX_DOMAIN]` — твой домен в hex

Для расчёта hex домена:
```bash
echo -n "your.domain" | xxd -ps
# например для tg.limbovpn.ru: 74672e6c696d626f76706e2e7275
```

Готовая ссылка из логов alexbers (третья из трёх, начинающаяся с `ee`):
```
https://t.me/proxy?server=your.domain&port=853&secret=ee89bea1ac525b5c7e58939d18a26e2ae574672e6c696d626f76706e2e7275
```

**Раздавать только эту**, FakeTLS-форму. Не раздавай голый 32-hex или dd-форму — у них хуже маскировка.

## Шаг 15. Финальная проверка

1. **Snapshot ключевых параметров с сервера:**
   ```bash
   echo | openssl s_client -connect 127.0.0.1:853 -servername your.domain 2>&1 | \
     grep subject
   # должно: subject=CN = your.domain
   ```

2. **Внешняя достижимость 5 стран:**
   ```bash
   curl -s "https://check-host.net/check-tcp?host=your.domain:853&max_nodes=5" \
        -H "Accept: application/json"
   # подожди 8 сек, потом по request_id из ответа:
   curl -s "https://check-host.net/check-result/REQUEST_ID" -H "Accept: application/json"
   ```

3. **Подключение в Telegram:**
   - Удали все прокси в Настройках Telegram (полная очистка).
   - Закрой клиент полностью (Диспетчер задач — для desktop).
   - Открой ссылку из Шага 14, тапни «Включить».
   - Через 5–10 секунд — зелёная галочка «Подключено».
   - Через **1–3 минуты** в верхушке списка чатов появится твой спонсорский канал.

4. **Счётчик в боте:**
   ```
   /myproxies
   ```
   Через минуту после первого реального коннекта счётчик перестанет быть нулевым.

## Управление ключами (что и зачем хранить)

| Ключ / артефакт | Где живёт | Зачем нужен | Можно ли публиковать |
|---|---|---|---|
| **Базовый 32-hex секрет** | `config.py` (USERS) | Идентификация пользователя | ❌ только в составе FakeTLS-формы |
| **AD_TAG** | `config.py` (AD_TAG) | Привязка к спонсорскому каналу | Привязан к проксу, утечка не критична |
| **Приватный ключ LE-сертификата** | Docker volume `caddy_data` | Подписывает TLS-handshake | ❌❌❌ никогда не публиковать |
| **A-запись DNS** | Регистратор домена | Маршрутизация клиентов на VPS | Публичная по природе |
| **FakeTLS-секрет (`ee...`)** | Логи alexbers, ссылка | Соединяет всё вместе для клиента | ✅ это и есть «приватная ссылка», которую раздаёшь |

При утечке базового секрета:
1. Сгенерируй новый (`head -c 16 /dev/urandom | xxd -ps`).
2. Обнови `USERS` в `config.py`.
3. `docker-compose restart alexbers`.
4. Удали старую запись в `@MTProxybot → /myproxies → Delete`, зарегистрируй новую с новым секретом.
5. Раздай новую ссылку. Старая мгновенно мертва.

## Если что-то не работает

### Логи alexbers: `default secret` или `default TLS_DOMAIN`
Конфиг не примонтирован.
```bash
docker exec mtproto-final cat /home/tgproxy/config.py
```
Если файл не такой как у тебя локально — проверь `volumes:` в docker-compose.yml.

### Логи alexbers: `cannot dial to telegram` / `i/o timeout`
Хостер режет исходящий к Telegram DC. Лекарство — менять VPS. Перепроверь Шаг 1 на новом.

### `Unable to connect to the Telegram server X (...)`
Один из middle-proxy IP не отвечает у твоего хостера. **Не критично**, если alexbers потом подключается к другим (проверь `cat /proc/net/tcp | awk '$4=="01"' | grep ":22B8"`).

### Telegram-клиент: «Недоступен» / «Подключение…» вечно
- Удали ВСЕ прокси в настройках, перезапусти приложение, тапни ссылку заново.
- Проверь IPv6 (`ping -6 your.domain` с клиента — если зависает, выключи IPv6 на адаптере).
- Антивирус с TLS-инспекцией (Касперский, Windows Defender enterprise) может ломать FakeTLS.

### Прокси работает, но через VPN — не подключается
Маршрут через VPN-провайдера упирается в blackhole у твоего/его хостера. Лечение: split-tunneling в VPN-клиенте (исключить твой домен/IP из туннеля), либо смена одного из хостеров.

### Текст ходит, медиа висит
Хостер блэкхолит часть Telegram DC IP. Точечно проверь весь диапазон `91.108.56.0/24` (см. Шаг 1) — если хоть один таймаутит, меняй хостинг.

### Спонсорский канал не появляется
По убыванию вероятности:
1. **Канал не привязан к новому прокси в боте** — самый частый случай. Возвращайся в Шаг 12.
2. AD_TAG не записан в `config.py` или контейнер не перезапущен после правки.
3. Регистрация прокси в боте только что прошла — подожди 5–10 минут на синхронизацию.
4. Telegram-клиент кеширует — полностью выгрузи и открой заново.
5. alexbers не может подключиться к Telegram middle-proxies на 8888 — проверь Шаг 1 на хостере.

### LE-сертификат не выдаётся (Caddy зависает на acquiring lock)
- Порт 80 не открыт наружу (нужен для ACME-challenge). Проверь у хостера.
- DNS не указывает на этот сервер (LE проверяет, что домен правда здесь).
- Rate-limit от LE (если уже было 5 неудачных попыток за час). Подожди час или используй `acme.zerossl.com` через настройку Caddy.

## Несколько IP для отказоустойчивости

DNS round-robin: добавь в DNS несколько A-записей `your.domain` на разные IP. Telegram-клиент при недоступности одного попробует следующий.

Варианты:
- **Допы IP у того же хостера** (€1–3/мес у Hostkey/Hetzner). Прописываются в `/etc/netplan/*.yaml`. alexbers слушает `0.0.0.0` — отвечает на любом IP без правок.
- **Несколько VPS в разных DC** (надёжнее). На каждом — идентичный `mtproxy/` каталог с тем же секретом и доменом. На каждом отдельно регистрируется LE-сертификат для домена (это допустимо — Caddy на каждом VPS получит свой). DNS round-robin балансит.

Перед каждым новым VPS прогоняй Шаг 1 (DC reach-check). Если хостер режет хоть один Telegram IP — он не годится.

## Безопасность и гигиена

- **Никогда не пушай `config.py` в публичные репозитории** — там USERS-секрет и AD_TAG.
- **Никогда не публикуй содержимое `caddy_data`** — там приватный ключ LE.
- Раздавай только полную FakeTLS-ссылку (`ee...`), не голый 32-hex.
- Регулярно `cd /root/mtproxy/src && git pull && cd .. && docker-compose up -d --build alexbers` — ловишь свежие фиксы движка.
- Разные пользователи — разные секреты в USERS-словаре, тогда сможешь отзывать индивидуально.
